
**项目名称：** 面向专业认证的毕业要求达成度统一计算平台

**目标业务：** 本科工程教育认证 / 师范类专业认证（OBE导向）

**评价方式：** 仅限基于学生成绩的“直接评价”

## 1. 业务概述与系统边界

本系统旨在为本科专业认证提供高精度、可追溯的数据计算支撑。系统将彻底摒弃繁杂的手工Excel计算，通过结构化的数据流转，实现从“底层考核点原始成绩录入”到“顶层毕业要求达成度输出”的全自动化计算。

**系统边界约束：**

- **适用层级：** 仅服务于本科生的专业认证数据处理。
    
- **数据流向：** 平台上游数据（学生名单、原始成绩）不依赖外部系统接口，均由用户手动录入或通过 Excel 格式化模板导入。
    
- **功能剔除：** 不包含问卷等间接评价模块；不包含成绩分布真实性校验或异常预警机制；不引入任何智能化/AI辅助扩展功能。

系统采用基于角色的访问控制（RBAC），严格划分以下四类业务角色：

| **角色**     | **系统职责与数据操作权限**                                                       |
| ---------- | --------------------------------------------------------------------- |
| **系统管理员**  | 负责系统初始化配置；管理全局字典（如学年学期、专业库）；分配与管理其余三类角色的账号与基础权限。                      |
| **教务管理人员** | 负责导入专业培养方案（课程库）；管理教学班级与学生名单（通过 Excel 导入）；监控全专业计算进度；导出最终专业认证归档报告。      |

## 3. 核心计算引擎架构及计算触发机制

系统的核心价值在于其数据溯源能力。评价模型采用“宏观矩阵-微观大纲”相分离的双层权重设计，数据计算按以下三个层级自下而上严格传导：

### 3.1 第一级：课程目标级达成度计算（基于考核点得分）

主讲教师将成绩细化至具体考核点（如“期末卷第1题”）。针对学生 $i$ 和课程目标 $j$：

$$C_{ij} = \frac{\sum (\text{支撑目标 } j \text{ 的考核点实际得分})}{\sum (\text{支撑目标 } j \text{ 的考核点目标满分})}$$

班级课程目标 $j$ 的达成度 $\bar{C_j}$ 为全体学生 $C_{ij}$ 的算术平均值。

### 3.2 第二级：课程级达成度计算（结合大纲内部贡献权重）

在单门课程内部，课程对某一毕业要求指标点 $k$ 的达成度 $E_k$，通过该课程各目标的达成度与其**大纲内部贡献权重 ($w$)** 计算得出：

$$E_{k} = \sum_{j} \left( \bar{C_j} \times w_{jk} \right)$$

_系统约束规则：_ 针对同一个指标点 $k$，相关联的各课程目标贡献权重之和必须满足 $\sum w_{jk} = 1.0$。

### 3.3 第三级：专业级达成度计算（结合宏观总支撑权重）

专业整体在某一指标点 $k$ 上的最终达成度 $G_k$，由所有支撑该指标点的课程达成度 $E_k$ 与其在**总矩阵中的支撑权重 ($W$)** 计算得出：

$$G_{k} = \sum \left( E_{k} \times W_{c} \right)$$

_系统约束规则：_ 全专业内，所有支撑同一指标点 $k$ 的相关课程，其总支撑权重之和必须满足 $\sum W_{c} = 1.0$。

### 3.4 计算触发机制：系统的计算链路在执行时分为两个独立的触发阶段：

- **阶段一（教师级触发）：** 针对第 3.1 节（目标级）与第 3.2 节（课程级）。由课程主讲教师在完成单科成绩导入后独立触发计算，结果仅锁定本课程数据。
    
- **阶段二（全局级触发）：** 针对第 3.3 节（专业级）。属于全局归集操作，需由专业负责人或教务管理员在所有先决课程完成阶段一计算后，统一触发执行。

## 4. 详细功能需求说明

### 4.1 基础与宏观数据管理 (模块 A)

- **A-1 全局数据字典设定：** 管理员配置学年、学期、学院、专业信息。
    
- **A-2 毕业要求管理 (专业负责人)：** 结构化录入毕业要求主体及细化的二级指标点（如 1.1, 1.2...）。
    
- **A-3 课程体系与名单导入 (教务管理员)：** 提供标准化 Excel 模板，批量导入全专业课程清单以及各教学班级的学生学号与姓名。
    
- **A-4 宏观支撑矩阵配置 (专业负责人)：**
    
    - 在二维网格界面中勾选课程与毕业要求指标点的支撑关系。
        
    - 设定每门课程对各个指标点的“总支撑权重 ($W$)”。
        
    - 系统需具备实时校验功能：按列（即每个指标点）实时求和，权重总和不为 `1.0` 时标红并禁止提交生效。
        

### 4.2 课程大纲与微观映射管理 (模块 B)

- **B-1 课程目标设定 (主讲教师)：** 定义本课程的若干教学目标（支持富文本/纯文本描述）。
    
- **B-2 内部权重分配 (主讲教师)：**
    
    - 系统自动读取模块A中的宏观矩阵，**仅向教师透出**本课程所负责支撑的毕业要求指标点。
        
    - 教师设定每个课程目标对这些指标点的“内部贡献权重 ($w$)“。
        
    - 系统按指标点维度进行横向校验，权重之和必须等于 `1.0`。
        
- **B-3 考核点细分与映射 (主讲教师)：**
    
    - 教师在系统中拆分具体考核点（如：平时作业1、实验报告2、期末试卷大题3）。
        
    - 录入每个考核点的满分分值，并在下拉菜单中将其绑定至对应的唯一的“课程目标”。
        

### 4.3 成绩录入与直接评价系统 (模块 C)

- **C-1 动态模板生成与导出：** 教师完成模块 B 后，系统根据其定义的考核点，自动生成带有复合表头的 Excel 成绩录入空模板（表头包含：学号、姓名、考核点1、考核点2...）。
    
- **C-2 原始成绩提交：** 教师将 Excel 数据回导至系统。系统提供类似电子表格的 Web 前端预览界面，支持简单的数据查缺补漏操作。
    
- **C-3  课程级达成度一键计算 (主讲教师)：**
	- 教师确认原始成绩无误后点击计算。
    - 系统按引擎规则，依次计算本课程的“目标级达成度”**以及**“课程级毕业要求指标点达成度”。
    - 计算完成后，系统将上述两级结果持久化至数据库，并将该课程的成绩单状态变更为“已锁定”。
     
- **C-4 专业级全局达成度计算 (专业负责人 / 教务管理员)：**

	- **前置校验：** 系统需提供一个宏观看板，展示某届学生/某学年涉及的所有支撑课程的计算状态（已锁定/未提交）。
	- **全局计算：** 当相关支撑课程全部计算完毕并锁定后，专业负责人或教务管理员可手动点击“执行专业级计算”。
	- **数据汇聚：** 系统提取各课程的“课程级指标点达成度”，结合宏观支撑矩阵的总权重，汇总计算出该专业各项毕业要求指标点的最终达成度，并持久化结果以供报表生成模块调用。
### 4.4 报表生成与底稿导出 (模块 D)

- **D-1 课程级评价报表：** 主讲教师可导出《课程目标达成情况评价表》PDF/Excel（包含班级单项平均分、各目标达成度明细）。
    
- **D-2 专业级报告与台账：** 教务管理员/专业负责人可导出全专业评价雷达图，以及支持认证专家查阅的“穿透式”Excel数据台账（展示从专业指标点层层向下追溯至某门课程、某个目标的得分明细）。

## 5. 非功能需求与技术规范建议

### 5.1 UI/UX 界面设计规范

- **视觉风格：** 严格遵循理性极简主义。摒弃不必要的渐变、阴影与动画。使用纯色背景、深色无衬线字体，利用留白与极细的边框线作为信息分割。
    
- **操作体验：** 矩阵配置与成绩校验界面需采用高数据密度的工业级数据表格设计（DataGrid），表头需支持冻结锁定，行列关联关系需通过高亮十字准星辅助用户视觉定位。
    

### 5.2 推荐技术选型方案 (多生态架构路线)

为确保系统具备高精度的数据聚合计算能力、复杂关系校验以及高并发的大型 Excel 解析能力，系统架构必须建立在强类型语言与成熟的企业级框架之上。研发团队可根据自身的技术栈储备，在以下几条主流技术路线中进行选型：

**路线一：Java 敏捷企业级生态（国内高校教务系统主流）**

- **后端支撑：** 采用 Java (JDK 17+) 配合 **Spring Boot 3** 框架构建微服务或单体架构。利用 MyBatis-Plus 或 Spring Data JPA 处理复杂的双层权重矩阵校验与持久化。
    
- **数据流处理：** 采用 Alibaba EasyExcel 或 Apache POI，应对百人级、多考核点复合表头模板的高效生成与内存安全解析。
    
- **前端展示：** **Vue 3** 配合 Element Plus / Ant Design Vue，构建数据响应迅速的后台 SPA。
    

**路线二：.NET 高性能工程化生态（原方案推荐）**

- **后端支撑：** 采用 C# 基于 **ASP.NET Core 8.0** Web API 构建 RESTful 服务。利用 C# 原生的 LINQ 强大集合操作能力，以极低的代码复杂度处理多维矩阵权重的聚合运算。
    
- **数据流处理：** 采用 EPPlus 或 ClosedXML 进行流式 Excel 读写。
    
- **前端展示：** **Blazor WebAssembly**（实现前后端 C# 语言统一，极致强类型校验）或主流的 Vue.js/React 单页应用。
    

**路线三：全栈 TypeScript 现代化路线（前后端语言统一）**

- **后端支撑：** 采用 **NestJS** (基于 Node.js)。它提供了严格的 TypeScript 类型约束与类似 Spring 的依赖注入（DI）架构，能够在保持前端敏捷性的同时，支撑严谨的后端业务逻辑与路由鉴权。
    
- **前端展示：** **React 18** 配合 Ant Design，利用 React 的高阶组件处理庞大复杂的 DataGrid（数据网格）性能优化与动态表头渲染。
    

**路线四：Python 数据处理与敏捷开发生态（极致的矩阵运算优势）**

- **后端支撑：** 采用 **FastAPI** 框架。其原生的 Pydantic 模型能提供极强的数据输入校验。针对双层权重矩阵的达成度计算，可直接引入 **NumPy** 或 **Pandas**，将复杂的嵌套循环聚合转化为高效的“向量化矩阵运算”，大幅提升海量成绩数据的计算性能。
    
- **数据流处理：** 采用 `pandas` 结合 `openpyxl` / `xlsxwriter`，极速实现复杂表头 Excel 模板的解析、生成与数据透视。
    
- **前端展示：** **Vue 3** 或 React，通过标准 RESTful 接口与后端交互。
    

**通用数据存储层（跨技术栈要求）：**

- **关系型数据库：** 必须选用强事务支持的关系型数据库，首选 **PostgreSQL 15+** 或 **MySQL 8.0+**。
    
- **底层优化要求：** 因计算链路涉及“考核点 $\to$ 目标 $\to$ 指标点”的多表深度联表查询，需针对 `CourseID`、`ObjectiveID` 和 `IndicatorID` 建立联合索引，并严格配置事务隔离级别（如 Read Committed 或 Repeatable Read）以支持“成绩锁定”的并发安全机制。
    

### 5.3 事务完整性与权限安全

- **计算锁定机制：** 成绩导入并触发计算后，该班级的原始成绩集和计算结果即被系统“锁定”。主讲教师失去修改权限。如确需勘误，需向教务管理员提交工单，由管理员解锁。
    
- **事务隔离：** 多名教师同时上传成绩或计算时，数据库层面需采用合适的事务隔离级别，防止数据幻读或死锁。
